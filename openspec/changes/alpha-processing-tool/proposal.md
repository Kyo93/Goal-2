# Proposal: Manual Alpha Processing Workflows for ISP Report Automation

Generated by: cm-planning
Date: 2026-06-05

## Why We're Doing This

The Alpha Knowledge Expert and Super Agent validation has passed. The current PoC proves that Alpha can answer from the generated Markdown package without confusing raw Zabbix alerts with confirmed incidents.

The next goal is to reduce manual refresh risk and split the raw-source processing into independently buildable Alpha steps. Instead of connecting Alpha to this repository through an API, the workflows will be built manually inside Alpha using the code and contracts stored in this repo.

## Qualified Problem

**For:** IT Operations users maintaining recurring ISP reports
**Who:** need to refresh reports from raw exports
**The:** current package-generation step is local and manual
**That:** should become a repeatable manual Alpha workflow built from deterministic processor nodes
**Unlike:** direct raw-file RAG upload
**Our approach:** copy source-specific deterministic processors into Alpha Code Executor/workflow nodes, then run a connector/correlation step with a strict validation gate before Knowledge Expert refresh

## Scope

### In Scope

- Define Alpha workflow processing contracts around the existing source adapters.
- Accept raw incident, Zabbix, issue-report, and optional master-data files.
- Process the three raw sources as separate flows.
- Connect the three canonical outputs through recurrence, correlation, validation, and package rendering.
- Generate the existing Alpha knowledge package format.
- Return validation status, source profile, warnings, and uploadable Markdown outputs.
- Block Knowledge Expert refresh when validation status is not `PASS`.
- Keep all counting, grouping, correlation, and reconciliation inside deterministic code.
- Treat this repository as the build kit/source of truth, not as an API service that Alpha calls.

### Out of Scope

- Letting the Super Agent calculate KPIs directly from raw files.
- Changing the package schema without a migration note.
- Automatic incident creation or operational ticket mutation.
- Alpha connecting to this repository through API/external-tool integration.
- Replacing the existing Knowledge Expert/Super Agent behavior.
- Treating unconfirmed master data as factual mapping.

## Recommendation

Implement the build in two phases:

1. Keep the local converter and processor-node files as verification/copy baselines.
2. Manually build equivalent Alpha workflows from the processor-node code and documented contracts.
3. Connect those Alpha workflows to each other inside Alpha; do not require Alpha to call this repo.

This keeps the proven package behavior intact while keeping the Alpha build practical and under manual control.

## Success Criteria

- Same raw inputs produce the same record IDs and package content as the current converter, except generated timestamps.
- Incident, Zabbix, and ticket workflows can be run independently on Alpha.
- The connector workflow can build the package from processed source JSON without re-reading raw source files.
- Validation report must pass before Alpha is allowed to ingest refreshed Markdown.
- Alpha Super Agent still passes the six validation prompts after a refreshed package.
- Each Alpha workflow response clearly reports `PASS`, `FAIL`, warnings, counts, and generated Markdown/JSON outputs.
