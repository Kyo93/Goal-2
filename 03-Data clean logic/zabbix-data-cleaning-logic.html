Zabbix Data Cleaning Logic

Tài liệu mô tả cách lấy và xử lý dữ liệu Zabbix cho Alpha Intelligent theo hướng chỉ giữ các monitoring signal có khả năng gây lỗi vận hành, giảm nhiễu từ full log, và tạo output ổn định cho workflow Clean Rawdata - Zabbix.

1. Quyết định kiến trúc

Alpha không connect trực tiếp đến Zabbix nội bộ. Zabbix API pull chạy ở máy local/VPN có quyền truy cập Zabbix. Kết quả là file CSV/XLSX đã normalize, sau đó upload vào Alpha để process tương tự ISP Incident Report.
1. Local/VPN Gọi Zabbix API bằng token. Không ghi token vào source code hoặc tài liệu.
2. Pull events Lấy trigger problem events bằng event.get, không lấy raw history/log.
3. Impact filter Chỉ giữ Tier 1, Tier 2 và context tối thiểu của Tier 3.
4. Upload file Upload CSV/XLSX normalized vào Alpha workflow.
5. Alpha process Tạo zabbix_alerts.json, alert_patterns.json và Markdown summary.

2. Tại sao không lấy full log?

Vấn đề

Zabbix có rất nhiều event lặp, recovery, probe timeout ngắn, agent health và resource warning.

Rủi ro

Nếu upload full log, SuperAgent dễ retrieve nhầm fragment và trả lời không nhất quán.

Giải pháp

Pull problem events, filter impact-only, sau đó group theo pattern để trả lời theo hướng vận hành.

Không dùng history.get cho use case này. History data phù hợp cho metric trend, không phù hợp để build operational incident context. Use case hiện tại cần trigger problem events có thời điểm bắt đầu, recovery, severity, host, site và signature.

3. Nguồn dữ liệu cần lấy

Nguồn Nên dùng Không nên dùng Lý do
Zabbix API event.get với trigger problem events history.get, raw log quá rộng Cần problem/recovery timeline, không cần metric samples dày đặc.
Severity Average,High,Disaster Information/Warning mặc định Giảm noise nhưng vẫn giữ Average để không bỏ lỡ network/firewall đặt severity thấp hơn.
Host group All VNM hoặc group vận hành cần phân tích Toàn bộ Zabbix không phân scope Giữ dữ liệu đúng phạm vi report.
Time range Theo tháng hoặc cửa sổ user hỏi Không giới hạn thời gian Query theo thời gian cần count chính xác và có thể tái lập.

4. Tier lọc impact-only

Tier Nhóm Ví dụ signature Quyết định Ý nghĩa vận hành
Tier 1 Network reachability unavailable by icmp ping, interface down, firewall is down, packet loss, latency, BGP/VPN/tunnel down Giữ luôn Có thể phản ánh mất kết nối, thiết bị down, path flap hoặc ISP/uplink issue.
Tier 2 HTTP monitoring git http monitoring, jira http monitoring, itcenter http monitoring, confluence http monitoring Giữ nếu duration_seconds >= 300 hoặc severity_id >= 4. Hữu ích nhưng dễ là threshold/probe timeout; cần group và đối chiếu incident/ticket.
Tier 3 Context kỹ thuật tối thiểu active checks are not available, host has been restarted, uptime Giữ làm context Không chứng minh user impact, nhưng giúp điều tra host/monitoring health.
Tier 3 Resource/capacity high cpu utilization, memory, disk, filesystem space Giữ nếu duration_seconds >= 1800 hoặc severity_id = 5. Tránh ngập noise từ resource warning ngắn; chỉ giữ signal đủ nặng/kéo dài.
Tier 1/3 Power/UPS UPS, battery, input frequency, power Giữ luôn Có thể ảnh hưởng facility/site, cần giữ để đối chiếu incident hoặc ticket.

5. Logic trong script local pull

File script: scripts/zabbix_data_pull_tool.py. Chế độ impact-only bật bằng flag --impact-only.

Command mẫu

cd C:\Users\Ocean\Documents\VibeCode\Goal2\Goal-2

$env:ZABBIX_TOKEN = "<token>"

python scripts\zabbix_data_pull_tool.py `
  --month 2026-05 `
  --host-group "All VNM" `
  --severities "Average,High,Disaster" `
  --impact-only `
  --output-csv 02-Output\zabbix_exports\zbx_problems_impact_All_VNM_2026-05.csv

Pseudocode lọc impact-only

family = classify(problem_signature, host, component, domain)

if family in ["Network reachability", "Power/UPS"]:
    keep

elif family == "HTTP monitoring":
    keep if duration_seconds >= 300 or severity_id >= 4

elif family in ["Zabbix agent/active check", "Host restart/uptime"]:
    keep as technical context

elif family == "Resource/capacity":
    keep if duration_seconds >= 1800 or severity_id == 5

else:
    drop
Token handling: không hardcode token trong file HTML, script, hoặc OpenSpec. Dùng environment variable ZABBIX_TOKEN hoặc secret parameter khi chạy local.

6. Output CSV để upload Alpha

File CSV output là normalized data, không phải full log. Đây là input cho workflow Clean Rawdata - Zabbix trên Alpha.

Field Ý nghĩa Dùng cho
event_id, trigger_id ID từ Zabbix Deduplicate và evidence reference.
host, host_id, site_code Host và site liên quan Query theo site và mapping timeline.
problem_raw, problem_signature Tên alert gốc và signature đã normalize Group pattern, phân loại family.
started_at_local, recovered_at_local Thời điểm phát sinh và recovery Filter theo khoảng thời gian user hỏi.
severity, severity_id, duration_seconds Mức độ và thời lượng Filter impact-only và priority.
domain, component, scope, tags_json Tag/context từ Zabbix Phân loại network, power, service, agent, resource.

7. Logic trong Alpha Zabbix Processor

File code: docs/alpha-zabbix-processor-node.py. Node này nhận zabbix_export_file hoặc zabbix_export_body.

Output JSON

  • zabbix_alerts.json: từng alert/event đã normalize.
  • alert_patterns.json: group theo host + problem signature.
  • summary: count, duplicate, rejected, upload readiness.

Field pattern bắt buộc

  • pattern_family
  • investigation_priority
  • signal_assessment
  • why_it_matters

Phân loại pattern_family

pattern_family Rule nhận diện Priority mặc định
Network reachability ICMP, unavailable, unreachable, down, packet loss, latency, gateway, VPN, BGP HIGH nếu firewall/switch/down hoặc lặp; MEDIUM nếu đơn lẻ.
HTTP monitoring http monitoring MEDIUM nếu alert_count >= 10, ngược lại LOW.
Zabbix agent/active check active checks, zabbix agent, nodata LOW
Host restart/uptime restarted, uptime LOW
Power/UPS UPS, battery, frequency, power HIGH
Resource/capacity CPU, memory, disk, filesystem, space LOW

8. Cách SuperAgent nên hiểu Zabbix

Loại evidence Cách diễn giải Không được nói
Zabbix-only Monitoring signal hoặc technical context. Không gọi là confirmed incident.
Network reachability Ưu tiên điều tra vì có thể liên quan connectivity/device/path issue. Không tự kết luận user impact nếu không có ticket/incident.
HTTP monitoring Probe/threshold signal; cần group theo pattern và xem có repeated/multi-host không. Không coi mọi HTTP probe timeout là service outage.
Agent/restart/resource Context kỹ thuật cho host/monitoring health. Không dùng làm proof RCA cho fiber/cable/routing incident.
