ITCenter Ticket Data Cleaning Logic

Tài liệu này giải thích script local đang làm gì với raw export từ ITCenter: gom comment-level rows thành ticket evidence, giữ ticket như bằng chứng user impact, và tạo dữ liệu đầu vào cho workflow Clean Rawdata - ITcenter.

1. Vai trò trong pipeline

ITCenter ticket là direct user evidence, không phải confirmed incident. Ticket giúp trả lời câu hỏi có người dùng/bộ phận nào bị ảnh hưởng, có symptom gì, và có bằng chứng trực tiếp nào khớp với incident hoặc monitoring signal hay không.
1. Raw export Đọc các monthly CSV trong 01-RawData/Ticket/*.csv.
2. Validate cột Kiểm tra 14 cột ITCenter bắt buộc.
3. Parse comment Mỗi row là một ticket comment, có timestamp và ticket id.
4. Group ticket Gom nhiều comment rows theo itcenter.ticket.id.
5. Output evidence Tạo aggregate, index, và monthly Markdown 05_ticket_impact_YYYY_MM.md.

2. Local script đang dùng phần nào?

Phần Giá trị hiện tại Ý nghĩa

Adapter local load_issue_tickets() trong src/itops_alpha/converter.py Hàm đọc, validate, reject row lỗi, group comment thành ticket.
Sheet được đọc Untitled Discover session (1) ( Converter đọc worksheet đầu tiên. Workbook có thêm Sheet1 nhưng sheet này không được dùng.
Package output 05_ticket_impact.md, 05_ticket_impact_index.md, 05_ticket_impact_YYYY_MM.md Aggregate ticket evidence, index theo tháng, và monthly partitions để giảm file size cho Alpha.
Evidence priority ITCenter đứng sau ISP Incident Report, trước Zabbix Alert Ticket là bằng chứng trực tiếp về user impact, còn Zabbix chỉ là monitoring signal.

Raw export là comment-level, không phải ticket-level. Một ticket có thể có nhiều rows vì mỗi comment/chat message là một dòng riêng. Vì vậy script không đếm row thành ticket; script phải group theo itcenter.ticket.id.

4. Logic clean và group

Bước Script làm gì Kết quả
Read row Bỏ qua blank row, tăng input_rows cho row có dữ liệu. Chỉ tính comment rows thật.
Parse time Parse itcenter.ticket.created_at, itcenter.ticket.comment.created_at, và itcenter.ticket.comment.updated_at. Dùng ticket created time làm mốc impact chính; comment time làm activity/detail.
Check id Clean itcenter.ticket.id. Reject row nếu ticket id trống.
Group Dùng grouped.setdefault(ticket_id, ...) để gom comment rows cùng ticket. Một ticket id chỉ tạo một record.
Preserve refs Lưu source_refs cho tất cả comment rows thuộc ticket. Có audit trail quay lại raw row.
Summarize Join tối đa 3 comment đầu tiên thành comments_summary. Knowledge có context ngắn, không cần expose toàn bộ chat dài ở summary chính.

Pseudocode

for each nonblank row:
    ticket_created_at = parse_datetime(row["itcenter.ticket.created_at"])
    comment_at = parse_datetime(row["itcenter.ticket.comment.created_at"])
    comment_updated_at = parse_datetime(row["itcenter.ticket.comment.updated_at"])
    ticket_id = clean_text(row["itcenter.ticket.id"])

    if comment_at invalid:
        reject row
    if ticket_id empty:
        reject row

    ticket = grouped.setdefault(ticket_id, first_seen_ticket_metadata)
    ticket.source_refs.append("<csv-file>:row-<n>")
    ticket.source_files.add(csv_file_name)
    ticket.ticket_created_times.append(ticket_created_at)
    ticket.comment_times.append(comment_at)
    ticket.comment_updated_times.append(comment_updated_at)
    ticket.comments.append(comment_text)

for each ticket_id:
    ticket_created_at = min(ticket_created_times)
    first_comment_at = min(comment_times)
    last_comment_at = max(comment_times)
    last_activity_at = max(comment_times + comment_updated_times)
    evidence_started_at = ticket_created_at or first_comment_at
    partition_months = months from comment_times
    comment_count = len(comment_times)
    comments_summary = " | ".join(first 3 non-empty comments)
    impact_evidence = title
    evidence_label = "SOURCE FACT"

5. Output ticket record

Output field Cách tạo Dùng để làm gì
ticket_id Raw itcenter.ticket.id Định danh ticket.
source_refs Tất cả raw comment rows thuộc ticket Audit trail và citation.
ticket_created_at, evidence_started_at Earliest raw ticket created time; fallback về first comment nếu thiếu Mốc chính để xác định thời điểm user/business bắt đầu bị ảnh hưởng.
category, classification, title Lấy từ row đầu tiên gặp của ticket Mô tả vấn đề ticket.
entity_name, location_full_name, office_display, subclassification Lấy từ service recipient/location fields mới Tăng khả năng xác định scope/site/office bị ảnh hưởng.
business_unit, comment_author_domains Lấy từ service recipient và comment author email domain Cho biết affected business scope và nguồn trao đổi trong ticket.
source_files, partition_months Lấy từ tên CSV và tháng của comment activity Chia nhỏ Knowledge theo tháng nhưng vẫn giữ lineage đầy đủ.
first_comment_at, last_comment_at, last_activity_at Min/max timestamp của comment rows và comment update rows Dùng để match activity window với incident/Zabbix.
comment_count Số comment rows đã gom Reconciliation và độ dày bằng chứng.
comments_summary 3 comment đầu tiên, nối bằng | Context ngắn cho Knowledge.
impact_evidence Bằng title Signal chính để đọc user impact.
evidence_label SOURCE FACT Đánh dấu đây là dữ liệu nguồn, không phải inference.

7. Ticket được dùng để correlate như thế nào?

Trong connector local, ticket chỉ được coi là bằng chứng impact khi có dấu hiệu match trực tiếp. Script không tự biến mọi ticket cùng ngày thành incident evidence.

Use case Rule chính Ý nghĩa
Incident với ticket Ticket text/title/evidence chứa site code dạng token độc lập và time window overlap với incident buffer. Có thể nâng user impact từ NO_DIRECT_EVIDENCE lên CONFIRMED.
Zabbix với ticket Ticket phải match site code, overlap time, và có relevance text đủ gần với alert/incident context. Có thể tạo USER_IMPACT_SIGNAL, nhưng vẫn không biến Zabbix thành confirmed incident.
Không match ticket Không có direct site-explicit ticket khớp. User impact là UNKNOWN hoặc NO_DIRECT_EVIDENCE, không được kết luận là không có user affected.
Cẩn thận với suy luận quá tay. Ticket là evidence mạnh hơn Zabbix cho user impact, nhưng vẫn cần match site/time/relevance. Ticket không khớp trực tiếp thì chỉ là dữ liệu nền, không phải bằng chứng impact cho incident cụ thể.

8. Hướng build Alpha ITCenter node

Khi build workflow Clean Rawdata - ITcenter trên Alpha, nên mirror logic local trước, sau đó mới thêm file output giống ISP/Zabbix.

Input nên nhận

  • issue_report_files hoặc từng issue_report_body theo monthly CSV
  • source_label / file name cho mỗi partition
  • return_markdown = true cho bước upload Knowledge

Output nên trả

  • ticket_evidence_text hoặc JSON compact
  • ticket_evidence_result
  • markdown aggregate
  • partition_markdowns theo tháng
  • partition_index_markdown
  • knowledge_file
  • knowledge_filename = 05_ticket_impact.md, 05_ticket_impact_index.md, hoặc 05_ticket_impact_YYYY_MM.md
  • summary

Validation tối thiểu

summary.ok == true
summary.input_rows == 9525
summary.normalized_tickets == 2681
summary.rejected_comment_rows == 0
summary.ticket_partition_count == 3
summary.reconciliation_passed == true
summary.upload_ready == true
knowledge_file.name ends with .md
